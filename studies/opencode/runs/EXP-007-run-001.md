# EXP-007-run-001

Date: 2026-06-27

## Objective

Validate that the `Part -> TextPartDisplay -> Markdown / PacedMarkdown -> DOM` rendering segment can be tested with a minimal happy-dom component test, without full Playwright e2e.

## Test created

**File:** `packages/session-ui/src/components/message-part-dom.test.tsx`

**Structure:**
- No explicit happy-dom import (uses `--preload` instead).
- Uses `bun:test` (`describe`, `expect`, `test`).
- Uses `solid-js/web` `render()` (already a workspace dependency).

**What it covers:**

| Case | Component path | Condition |
|------|---------------|-----------|
| Completed assistant text part | `Part -> TextPartDisplay -> Markdown -> DOM` | `message.time.completed` is set |
| Incomplete (streaming) assistant text part | `Part -> TextPartDisplay -> PacedMarkdown -> Markdown -> DOM` | `message.time.completed` is absent |

**Providers mounted:**
- `MarkedProvider` (real, from `@opencode-ai/ui/context/marked`)
- `DataProvider` (real, from `../context/data`)

**Text target:**
```

- first item
- second item with `inlineCode`

[OpenCode](https://opencode.ai)
```

**DOM assertions:**
- `[data-component="text-part"]` exists
- `[data-component="markdown"]` exists
- `h2` textContent is `"Runtime heading"`
- `li` count is 2
- `code` textContent is `"inlineCode"`
- `a` href is `"https://opencode.ai"`

**Polling:** 2-second `waitFor` loop (10ms interval) to allow Solid effects + markdown parse.

## Execution attempts

### Attempt 1 — Bun not in PATH

**Status:** Resolved.

Bun 1.3.14 existed at `<USER_HOME>\AppData\Roaming\npm\bun.cmd` but `%APPDATA%\npm` was not in `$env:PATH`. Temporary prepend fixed it.

### Attempt 2 — Bun/Vite worker import incompatibility (first attempt)

**Status:** Resolved via `mock.module()` preload.

`bun test` failed with:
```
SyntaxError: Missing 'default' export in module './markdown-shiki.worker.ts?worker&url'
```

`?worker&url` is Vite-specific syntax. Bun's module resolver treats the query as part of the module specifier and cannot resolve it.

**Resolution:** Created `packages/session-ui/src/components/happydom.ts` as a preload that:
1. Reuses the app's happy-dom setup (`import "../../../app/happydom"`)
2. Calls `mock.module("./markdown-worker", ...)` to mock the worker module before any test import reaches it
3. Provides `globalThis.React = { createElement: h }` (imported from `solid-js/h`) so Bun's JSX transpiler delegates to Solid's `h` function

### Attempt 3 — SolidJS server build resolution

`--conditions=browser` flag was needed to resolve `solid-js` to the browser build instead of the server build. Without it, Solid throws `Client-only API called on the server side`.

### Attempt 4 — JSX transpilation (React not defined)

Bun's default JSX transpiler uses `React.createElement`. Since the components are Solid, `globalThis.React` was provided via `solid-js/h`.

## Limitations

- The worker mock means code-block syntax highlighting is not exercised (only inline markdown rendering is tested).
- The test uses `--conditions=browser` to select Solid's browser build, which is consistent with the app's runtime context.
- HappyDOM is not a full browser engine (no layout, no scripting), but it supports the DOM APIs needed for `Markdown` rendering.
